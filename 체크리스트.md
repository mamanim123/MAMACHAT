# MAMA Chat / Hermes Portable Agent Plan

## 1. 대시보드 1차 확정안

### 대화창
- 대화창은 화면 대부분을 차지한다.
- 사이드바는 마우스 호버 방식으로 작동한다.
- 평소에는 얇은 아이콘 바만 보이고, 마우스를 올리면 확장된다.

### 하단 입력 영역
- 입력필드 왼쪽에 스피커 / 마이크 아이콘을 둔다.
- 실행 버튼은 Enter 입력 시 바로 실행된다.
- Shift + Enter는 줄바꿈으로 사용한다.
- 실행 중에는 실행 버튼이 중지 버튼으로 바뀐다.
- 중지 버튼을 누르면 AI 응답, 백엔드 작업, 실행 중인 명령어를 모두 중지한다.
- 실행 버튼 옆에 계획모드를 둔다.

### 스피커 기능
- 스피커 클릭 시 마지막 AI 답변을 음성으로 읽는다.
- 음성 출력 중에는 스피커 주변이 빨간색으로 깜빡인다.
- 음성 출력 중 스피커를 다시 클릭하면 즉시 음성 출력을 중지한다.

### 마이크 기능
- 마이크 클릭 시 음성 인식을 시작한다.
- 음성은 텍스트로 변환되어 입력필드에 들어간다.
- 음성 인식 후 바로 실행하지 않고 사용자가 확인 후 실행한다.

## 2. 사이드바 확정안

### 창 단위 세션
- 사이드바는 메시지 단위가 아니라 채팅창/세션 단위로 표시한다.
- 각 창은 독립된 대화 기록과 메모리를 가진다.
- 이전 창을 클릭하면 해당 창에서 나눈 대화만 표시된다.
- 해당 창의 작업 맥락을 유지한 채 이어서 작업할 수 있다.
- 에이전트작업중 창을 닫고 나가도 백그라운드에서 작업을 계속하고 완료되면 화면상단에 크게 토스트로 알림을 준다.

### 즐겨찾기 / 삭제
- 각 창에는 즐겨찾기 별 아이콘이 있다.
- 빈 별을 누르면 노란색 별로 바뀌고 즐겨찾기에 등록된다.
- 즐겨찾기 된 창은 사이드바 상단에 고정된다.
- 즐겨찾기 된 창은 삭제할 수 없다.
- 삭제하려면 먼저 별표를 해제해야 한다.
- 삭제 버튼은 마우스 호버 시 표시한다.
- 삭제 버튼 클릭 시 확인창을 띄운다.

## 5. 저장 구조 확정

### 기본 방향
MAMA Chat / Hermes Portable Agent는 SQLite와 작업폴더 저장 방식을 함께 사용하는 하이브리드 구조로 설계한다.

### SQLite에 저장할 정보
SQLite는 앱 전체의 대화창 관리와 검색을 담당한다.

저장 항목:
- 창 목록
- 창 제목
- 창별 대화 메시지
- 즐겨찾기 여부
- 삭제 여부
- 최근 사용 시간
- 선택한 모델 정보
- 선택한 작업 폴더 경로
- 계획모드 상태
- 마지막 작업 요약
- 다음 작업 정보

### 작업폴더에 저장할 정보
선택한 작업폴더 안에는 `.mama-agent` 폴더를 만들고, 해당 프로젝트에 필요한 기억과 규칙을 저장한다.

예시:

```text
선택한작업폴더/
 ├─ .mama-agent/
 │   ├─ project_memory.md
 │   ├─ project_rules.md
 │   ├─ last_summary.md
 │   ├─ next_action.md
 │   └─ skills/
 └─ 실제 프로젝트 파일들
역할 분리

SQLite:

앱 전체 관리
대화창 목록
메시지 검색
즐겨찾기
삭제 상태
창 복원

작업폴더 .mama-agent:

프로젝트별 기억
프로젝트별 규칙
마지막 작업 요약
다음 작업 계획
해당 폴더 전용 스킬
장점
SQLite를 사용해 대시보드와 검색 속도를 안정적으로 유지할 수 있다.
작업폴더에 .mama-agent를 두면 폴더를 옮겨도 프로젝트 기억을 함께 가져갈 수 있다.
앱 전체 대화 관리와 프로젝트별 포터블 기억을 동시에 만족할 수 있다.
확정

창 목록, 메시지, 즐겨찾기, 삭제 상태, 모델 선택 정보는 SQLite에 저장한다.
선택한 작업폴더별 프로젝트 메모리와 규칙은 해당 폴더의 .mama-agent 폴더에 저장한다.
따라서 앱은 전체 대화 관리가 가능하고, 프로젝트 폴더는 포터블한 작업 기억을 가진다.

## 6. 모델 선택 구조

### 기본 방향
MAMA Chat / Hermes Portable Agent는 모델을 두 종류로 나누어 관리한다.

1. API 기반 대화 모델
2. 인증 기반 코딩 실행자

### 1. API 기반 대화 모델
API 기반 모델은 주로 저렴하거나 무료로 사용할 수 있는 모델을 사용한다.

용도:
- 일반 대화
- 간단한 코드 설명
- 문서 요약
- plan.md 정리
- 작업 계획 초안 작성
- 가벼운 리서치

특징:
- API Key 방식으로 연결한다.
- 비용이 적거나 무료인 모델을 우선 사용한다.
- 대화창의 기본 응답 모델로 사용한다.
- 모델은 나중에 OpenRouter, Gemini, Groq, Together, Hugging Face 등으로 확장할 수 있게 설계한다.
- 특정 무료 모델 이름은 고정하지 않고 설정 파일에서 바꿀 수 있게 한다.

### 2. 인증 기반 코딩 실행자
Claude Code와 Codex는 API 모델처럼 직접 호출하는 것이 아니라,
사용자가 로컬에서 인증한 CLI 또는 앱을 실행자로 연결한다.

용도:
- 코드베이스 분석
- 파일 수정
- 명령어 실행
- 테스트 작성
- 버그 수정
- 리팩터링
- 하네스 기반 검증 작업

특징:
- 사용자가 각 도구에 직접 로그인한다.
- 앱은 인증 정보를 직접 저장하지 않는다.
- 앱은 선택한 작업 폴더에서 Claude Code 또는 Codex 명령을 실행한다.
- 실행 결과, 로그, 수정 파일 목록을 대시보드에 표시한다.
- 위험한 작업은 계획모드 또는 사용자 승인 후 실행한다.

### UI 구조
상단바에는 다음과 같이 표시한다.

```text
대화 모델: 무료 API 모델 ▼
코딩 실행자: Codex ▼

또는 간단 모드에서는 다음처럼 표시할 수 있다.

모델: 무료 API 모델 / Claude Code / Codex

하지만 내부 구조는 반드시 대화 모델과 코딩 실행자를 분리한다.

실행 흐름 예시
사용자 입력
→ 무료 API 모델이 요청을 이해하고 계획 작성
→ 코드 수정이 필요한 경우 Claude Code 또는 Codex 실행자 선택
→ 선택한 작업 폴더에서 실행
→ 결과 로그 수집
→ 무료 API 모델이 결과를 요약
→ 창별 메모리와 작업폴더 .mama-agent에 요약 저장
보안 원칙
API Key는 .env 또는 안전한 로컬 설정에만 저장한다.
Claude Code / Codex 로그인 정보는 앱이 직접 저장하지 않는다.
선택한 작업폴더 밖의 파일은 수정하지 못하게 한다.
삭제, 배포, 대량 수정, 외부 전송 작업은 사용자 승인 후 실행한다.
실행 중에는 언제든지 중지 버튼으로 작업을 중단할 수 있어야 한다.
확정

기본 대화와 가벼운 작업은 무료 API 모델을 사용한다.
코드 수정과 프로젝트 폴더 기반 작업은 Claude Code 또는 Codex를 인증 기반 실행자로 사용한다.
UI에서는 대화 모델과 코딩 실행자를 분리해서 선택할 수 있게 한다.

## 7. 영상 기반 필수 반영 기능 체크리스트

### 기본 방향
첨부한 Hermes Agent / Claude Code / Harness 영상에서 나온 유용한 기능은 설계 과정에서 빠뜨리지 않고 반영한다.
MAMA Chat은 단순 채팅 UI가 아니라, 기억하고 실행하고 개선되는 개인 작업 에이전트 대시보드를 목표로 한다.

### 1. 메모리 기능
- 사용자 전체 기억과 창별 기억을 분리한다.
- 사용자 취향, 기본 설정, 장기 목표는 전역 메모리에 저장한다.
- 각 창의 작업 맥락, 마지막 요약, 다음 작업은 창별 메모리에 저장한다.
- 프로젝트별 규칙과 작업 요약은 선택한 작업폴더의 `.mama-agent`에 저장한다.
- 오래 유지할 사실만 메모리에 저장하고, 임시 작업 상태나 API 키는 저장하지 않는다.

### 2. 세션 검색 기능
- 예전 채팅창을 검색할 수 있어야 한다.
- 특정 키워드로 과거 창과 대화 내용을 찾을 수 있어야 한다.
- 검색 결과를 클릭하면 해당 창으로 바로 이동한다.
- SQLite 기반으로 전체 대화 검색을 지원한다.

### 3. 스킬 기능
- 반복되는 작업은 skill 형태로 저장한다.
- 스킬은 Markdown 파일로 관리한다.
- 작업을 여러 번 반복하면 에이전트가 스킬 생성을 제안할 수 있어야 한다.
- 기존 스킬을 수정하고 개선할 수 있어야 한다.
- 스킬은 필요한 순간에만 불러와서 컨텍스트가 불필요하게 커지지 않게 한다.

### 4. 예약 작업 / Cron 기능
- 사용자가 자연어로 예약 작업을 만들 수 있어야 한다.
- 예: 매일 아침 브리핑, 매주 GitHub 트렌드 조사, 주간 작업 요약
- 예약 작업은 목록으로 확인할 수 있어야 한다.
- 예약 작업은 추가, 일시중지, 삭제할 수 있어야 한다.
- 예약 작업 결과는 지정한 창 또는 알림 채널로 전달한다.

### 5. 음성 기능
- 마이크 입력을 텍스트로 변환한다.
- 스피커 출력으로 AI 답변을 들을 수 있게 한다.
- 음성 출력 중에는 스피커 아이콘 주변이 빨간색으로 깜빡인다.
- 스피커를 다시 클릭하면 음성 출력을 중지한다.

### 6. 모델 비용 관리
- 모든 작업을 비싼 모델로 처리하지 않는다.
- 기본 대화와 가벼운 작업은 무료 또는 저렴한 API 모델을 사용한다.
- 복잡한 코드 작업은 Claude Code 또는 Codex 실행자를 사용한다.
- 작업 성격에 따라 모델을 바꿀 수 있어야 한다.
- 나중에 비용/사용량 표시 기능을 추가할 수 있게 설계한다.

### 7. 보안 기능
- API Key와 토큰은 메모리에 저장하지 않는다.
- `.env` 또는 안전한 로컬 설정에만 저장한다.
- 선택한 작업폴더 밖의 파일은 수정하지 못하게 한다.
- 위험한 작업은 실행 전 사용자 승인을 받는다.
- 대량 삭제, 배포, 외부 전송, 시스템 변경은 확인 후 실행한다.
- 새 스킬 설치 전에는 출처와 위험성을 확인한다.

### 8. 작업 투명성
- 에이전트가 어떤 도구를 사용했는지 볼 수 있어야 한다.
- 실행한 명령어, 수정한 파일, 발생한 에러를 작업 로그에 표시한다.
- 작업 로그는 기본적으로 접어두고 필요할 때 펼친다.
- 실행 중에는 언제든지 중지할 수 있어야 한다.

### 9. 계획모드 / 검증 루프
- 코드 수정이나 파일 변경 전에는 계획모드를 사용할 수 있어야 한다.
- 계획모드에서는 어떤 파일을 확인하고 수정할지 먼저 보여준다.
- 실행 후에는 결과 요약, 수정 파일, 다음 작업을 남긴다.
- 반복된 실패는 retro 또는 memory에 기록한다.
- 큰 작업은 Plan → Execute → Verify → Retro 흐름으로 처리한다.

### 10. 멀티 에이전트 확장
- 처음에는 하나의 개인 에이전트로 시작한다.
- 나중에 필요하면 역할별 에이전트를 분리한다.
- 예: 리서치 에이전트, 코딩 에이전트, 브리핑 에이전트
- 에이전트별로 메모리, 도구, 권한, 키를 분리할 수 있게 설계한다.
- 처음부터 복잡한 멀티 에이전트 구조로 가지 않고 확장 가능성만 열어둔다.

### 11. 메시징 플랫폼 연동
- 1차 목표는 웹 UI 중심으로 만든다.
- 이후 Telegram, Discord, Slack 같은 외부 채널 연동을 고려한다.
- 외부 채널에서도 같은 에이전트의 메모리와 스킬을 사용할 수 있게 설계한다.

### 12. MCP / 외부 도구 연동
- GitHub, 파일 시스템, 브라우저 자동화, 검색 도구를 나중에 연결할 수 있게 한다.
- 처음부터 많은 MCP를 붙이지 않고, 필요한 것부터 단계적으로 추가한다.
- MCP 연결 시 권한과 보안을 우선 점검한다.

### 13. 자기진화 루프
- 작업 경험이 메모리, 스킬, 검색 가능한 기록으로 남아야 한다.
- 같은 작업을 반복하면 더 잘 처리하도록 스킬을 개선한다.
- 사용자의 피드백을 받아 스킬과 규칙을 수정한다.
- 실패한 작업은 다음 작업의 개선 규칙으로 남긴다.

### 확정
이 체크리스트는 앞으로 기능을 추가할 때 기준으로 사용한다.
새 기능을 설계할 때마다 이 목록과 비교해서 빠진 기능이 있는지 확인한다.

## 9. 실행 모드 / 위험작업 승인 구조

### 기본 방향
MAMA Chat / Hermes Portable Agent는 작업 실행 방식을 수동 / 반자동 / 완전자동 3단계로 제공한다.
단, 어떤 모드에서도 위험작업은 반드시 사용자 승인을 받아야 한다.

---

### 1. 수동 모드

수동 모드는 가장 안전한 실행 방식이다.

동작 방식:
- 에이전트가 먼저 작업 계획을 제시한다.
- 사용자가 각 단계를 승인해야 실행한다.
- 승인되지 않은 작업은 실행하지 않는다.

용도:
- 처음 사용하는 프로젝트
- 중요한 파일 수정
- 구조 변경
- 에러 원인 분석
- 민감한 작업

---

### 2. 반자동 모드

반자동 모드는 기본 추천 모드다.

동작 방식:
- 에이전트가 먼저 작업 계획을 제시한다.
- 사용자가 계획을 승인하면 승인된 범위 안에서 자동으로 진행한다.
- 중간에 위험작업이 나오면 즉시 멈추고 사용자 승인을 요청한다.

용도:
- 일반적인 코드 수정
- UI 수정
- 버그 수정
- 문서 정리
- plan.md 수정
- 빌드 확인

기본값:
- 실행 모드의 기본값은 반자동 모드로 한다.

---

### 3. 완전자동 모드

완전자동 모드는 일반 작업을 최대한 자동으로 처리하는 모드다.

동작 방식:
- 일반 작업은 계획, 실행, 검증, 요약까지 자동으로 진행한다.
- 단, 위험작업은 무조건 멈추고 사용자 승인을 요청한다.
- 완전자동은 위험작업 자동 승인을 의미하지 않는다.

용도:
- 문서 정리
- 코드 포맷팅
- 단순 리팩터링
- 반복 점검
- 로그 분석
- 테스트 재실행
- 정해진 규칙 안의 반복 작업

---

### 4. 위험작업 승인 원칙

다음 작업은 수동 / 반자동 / 완전자동 어떤 모드에서도 반드시 사용자 승인을 받아야 한다.

- 파일 삭제
- 폴더 삭제
- Git reset
- Git clean
- 강제 push
- 배포 명령어 실행
- `.env` 수정
- API Key / 토큰 관련 작업
- 선택한 작업폴더 밖 접근
- 데이터베이스 초기화
- 대량 파일 변경
- 패키지 대량 설치
- 외부 서버로 파일 전송
- 권한 변경
- 시스템 설정 변경

---

### 5. UI 구조

상단바 또는 입력창 근처에 실행 모드 선택 드롭다운을 둔다.

```text

선택지:

수동

공간을 아끼기 위해 버튼형보다 드롭다운 방식을 우선 사용한다.

확정

실행 모드는 수동 / 반자동 / 완전자동 3단계로 제공한다.
기본값은 반자동 모드로 한다.
어떤 모드에서도 위험작업은 무조건 사용자 승인을 받아야 한다.
완전자동 모드는 일반 작업 자동화를 의미하며, 위험작업 자동 승인을 의미하지 않는다.

MAMA Chat / Hermes Portable Agent는 사용자가 언제든지 작업을 중지할 수 있어야 하며,
파일이 수정되는 작업은 되돌릴 수 있어야 한다.

---

- 중지 완료 후 stopped 상태로 변경

- AI 응답 스트리밍
- API 요청
- Claude Code 실행 프로세스

- 장기 실행 작업

이런 작업은 실행 전 반드시 승인받고,
가능하면 되돌리기 방법을 계획모드에서 먼저 제시한다.

10. Git 사용 시 처리

작업폴더가 Git 저장소인 경우에는 되돌리기 기능과 별도로 git diff를 보여줄 수 있다.

지원 기능:

수정 전 git status 확인
수정 후 git diff 표시
변경 파일 목록 표시
사용자가 원하면 백업 기반 되돌리기 실행

주의:

자동으로 git reset을 실행하지 않는다.
git reset, git clean은 위험작업으로 분류하고 반드시 사용자 승인을 받는다.
확정

실행 중지 기능과 되돌리기 기능은 1차 기능에 포함한다.
파일 수정 전에는 작업 단위 백업을 생성한다.
작업 완료 후에는 수정 파일 목록과 되돌리기 버튼을 제공한다.
되돌리기는 .mama-agent/backups/에 저장된 백업을 기준으로 수행한다.
위험작업은 모드와 관계없이 반드시 사용자 승인을 받아야 한다.

## 11. 사전검사 / 의존성 체크 / 안전 실행 게이트

### 기본 방향
MAMA Chat / Hermes Portable Agent는 파일을 수정하기 전에 반드시 사전검사를 수행한다.
단순히 백업 후 수정하는 것이 아니라, 수정 전에 프로젝트 상태와 위험 요소를 먼저 확인하고,
수정 후에는 검증을 실행하며, 문제가 생기면 되돌리기 또는 재시도를 선택할 수 있게 한다.

---

### 1. 사전검사 목적

사전검사는 에이전트가 파일을 수정하기 전에 현재 작업 환경을 확인하는 단계다.

목적:
- 현재 작업폴더가 맞는지 확인한다.
- 프로젝트 유형을 파악한다.
- 의존성 파일과 설치 상태를 확인한다.

## 12. 입력필드 상단 템플릿 / 스킬 버튼 구조

### 기본 방향
MAMA Chat / Hermes Portable Agent는 입력필드 바로 위에 자주 사용하는 작업 버튼을 배치한다.
사용자는 매번 긴 지시문을 직접 작성하지 않고, 버튼을 눌러 자주 쓰는 작업 지시문을 빠르게 입력할 수 있다.

---

### 1. UI 위치

템플릿 / 스킬 버튼은 입력필드 바로 위에 배치한다.

```text
[⚡ 작업 ▼] [⭐ 템플릿 ▼] [🧠 스킬 ▼] [실행모드: 반자동 ▼]

[분석] [버그 찾기] [개선] [주석 추가]
[리팩토링] [동작 검증] [의존성 점검] [plan 업데이트]

------------------------------------------------
| 에이전트 작업 지시 입력창                    |
------------------------------------------------
| 🔊  🎙                         [실행 / 중지] |
2. 템플릿 버튼 동작

템플릿 버튼을 클릭해도 바로 실행하지 않는다.
버튼을 누르면 해당 작업에 맞는 지시문이 입력필드에 자동으로 채워진다.

사용자는 입력된 지시문을 확인하고 수정한 뒤,
Enter 또는 실행 버튼으로 작업을 실행한다.

예시:

버그 찾기 버튼 클릭 시 입력필드에 자동 입력:

현재 선택한 작업폴더에서 버그 원인을 찾아줘.
먼저 사전검사와 의존성 체크를 하고,
관련 파일을 분석한 뒤 수정 전 계획을 제시해줘.
수정이 필요하면 백업 후 진행하고, 검증 결과까지 알려줘.
3. 1차 템플릿 버튼

초기 버전에서는 너무 많은 버튼을 넣지 않고, 자주 쓰는 핵심 기능만 제공한다.

1차 템플릿 버튼:

분석
버그 찾기
개선
주석 추가
리팩토링
동작 검증
의존성 점검
plan 업데이트
4. 템플릿과 스킬의 차이

템플릿과 스킬은 분리해서 관리한다.

템플릿 = 자주 쓰는 작업 시작 양식
스킬 = 실제로 성공한 작업 방식을 저장한 재사용 절차

예시:

템플릿:
버그 찾기

스킬:
MAMA Chat 프로젝트에서 JSX 문법 오류를 찾고 수정하는 검증된 절차

템플릿은 기본 제공되는 작업 양식이고,
스킬은 사용자가 실제 작업을 수행한 뒤 저장한 개인화된 작업 절차다.

5. 스킬 버튼

입력필드 위에는 템플릿 버튼과 별도로 스킬 버튼을 둔다.

[⭐ 템플릿 ▼] [🧠 스킬 ▼]

스킬 버튼을 누르면 저장된 전역 스킬 또는 현재 프로젝트 스킬을 선택할 수 있다.

스킬 종류:

전역 스킬
현재 작업폴더 전용 스킬

전역 스킬 예시:

Git diff 검토
plan.md 업데이트
빌드 에러 해결
의존성 점검

프로젝트 전용 스킬 예시:

MAMA Chat 대시보드 수정 절차
AgentPanel 오류 수정 절차
현재 프로젝트 빌드 검증 절차
6. 작업 완료 후 스킬화

작업이 완료되면 대화창 또는 작업 로그에 다음 버튼을 제공한다.

[되돌리기] [변경내용 보기] [스킬로 저장] [요약 저장]

스킬로 저장을 누르면 현재 작업 방식이 Markdown 스킬 파일로 저장된다.

스킬 저장 시 입력 항목:

설명
사용 조건
실행 순서
안전 규칙
검증 방법
되돌리기 방법
저장 위치

저장 위치:

전역 스킬
현재 프로젝트 스킬
7. 스킬 생성 제안

에이전트는 반복 가능성이 높은 작업이 끝나면 사용자에게 스킬 생성을 제안할 수 있다.

예시:

이번 작업은 반복 가능성이 높습니다.
이 작업 방식을 스킬로 저장할까요?

[스킬로 저장] [아니요]

단, 스킬은 사용자 승인 후에만 저장한다.

8. 스킬로 만들기 좋은 작업

스킬로 만들기 좋은 작업:

빌드 에러 분석 후 수정
Git diff 요약
React 컴포넌트 오류 수정
작업 후 last_summary.md 업데이트
폴더 구조 분석
사전검사 실행
동작 검증

스킬로 만들면 안 되는 작업:

단순 잡담
한 번만 쓰는 임시 작업
API Key나 토큰이 포함된 작업
검증 방법이 없는 작업
너무 특정 경로에만 묶인 작업
9. 확정

입력필드 바로 위에 템플릿 / 스킬 버튼 영역을 둔다.
템플릿 버튼은 자주 사용하는 작업 지시문을 빠르게 입력하기 위한 기능이다.
템플릿 버튼을 눌러도 바로 실행하지 않고, 입력필드에 지시문을 자동 입력한다.
사용자는 입력된 지시문을 수정한 뒤 Enter 또는 실행 버튼으로 실행한다.
템플릿과 스킬은 구분한다.
스킬은 성공한 작업 절차를 저장한 재사용 가능한 작업 방식이다.
작업 완료 후에는 스킬로 저장 기능을 제공한다.

## 13. 무료 모델 / Qwen 계열 모델 업데이트 추적

### 기본 방향
MAMA Chat / Hermes Portable Agent는 무료 또는 저비용으로 사용할 수 있는 모델을 주기적으로 확인하고,
현재 작업에 적합한 모델을 추천할 수 있어야 한다.

특정 모델 하나에 고정하지 않고,
Qwen 계열, OpenRouter 무료 모델, Gemini 무료 티어, Groq, Together, Hugging Face 등

- 모델별 장단점을 비교한다.
- 작업 유형에 맞는 모델을 추천한다.
- 무료 제공 기간이나 제한이 바뀌면 사용자에게 알려준다.
- 기본 대화 모델 후보를 자동으로 업데이트할 수 있게 한다.

### 2. 추적 대상

- Nous Portal 무료 모델

13. 크론 안전 원칙

크론은 자동 실행되므로 권한을 제한한다.

원칙:

크론은 plan.md를 자동 수정하지 않는다.
크론은 파일 삭제, 배포, git reset 같은 위험작업을 하지 않는다.
크론은 정보 수집, 분석, 후보 저장까지만 수행한다.
plan.md 반영은 사용자 승인 후에만 가능하다.
실패한 크론은 실패 로그를 저장한다.
출처가 부족한 정보는 더 조사 상태로 둔다.
API Key나 토큰은 크론 결과에 저장하지 않는다.
14. 기능 후보함과 크론의 관계

동작 흐름:

크론 실행
→ 새 기능 / 모델 / 도구 발견
→ 출처 링크 저장
→ GitHub 교차검색
→ 우선순위와 위험도 평가
→ 기능 후보함에 저장
→ 사용자가 채택 / 보류 / 폐기 결정
→ 확정된 기능만 plan.md 반영
확정

기능 후보함은 업데이트 감시 크론이 발견한 새 기능을 저장하고 관리하는 공간이다.
각 기능 후보는 출처 링크, GitHub 교차검색 결과, 우선순위, 난이도, 위험도, 비용 영향, 적용 가능성을 가진다.
기능 후보는 검토중 / 테스트 필요 / 채택 / 보류 / 폐기 / 더 조사 / plan.md 반영 후보 / plan.md 반영 완료 상태로 관리한다.
크론 관리 UI에서는 자동 작업 목록을 확인하고, 즉시 실행 / 일시중지 / 재개 / 수정 / 삭제 / 실행 로그 보기를 할 수 있다.
크론은 자동으로 plan.md를 수정하지 않고, 정보 수집과 후보 저장까지만 수행한다.
plan.md 반영은 사용자 승인 후에만 진행한다.
